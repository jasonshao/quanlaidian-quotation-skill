# 飞书报价卡片 MVP 设计

日期：2026-04-12

## 目标

在飞书机器人对话中，为全来店报价技能增加“少打字、以选择为主”的报价引导体验。

本次只做 MVP，目标是在不依赖飞书交互卡片回调的前提下，让用户通过“静态卡片 + 文本回复”完成标准报价所需信息收集，并复用现有报价引擎生成 PDF、Excel、JSON。

## 背景与约束

- OpenClaw 环境下，技能可以通过提示词和多轮对话完成字段收集，但飞书当前没有可用的卡片回调接入权限。
- 飞书机器人现有能力已支持发送文本消息和文件消息，代码基础在 `scripts/feishu_file_delivery.py`。
- 当前项目已有成熟的报价生成链路：
  - `scripts/build_quotation_config.py`
  - `scripts/run_openclaw_quotation.py`
- 标准自动报价范围已收敛为 `1~30 店`，`31 店及以上` 转人工。
- 希望用户尽量通过“看卡片 + 回复数字/名称”的方式完成录入，减少自由输入负担。

## 范围

### 本次包含

- 飞书会话中的多轮报价引导
- 静态飞书卡片渲染
- 数字序号 / 中文名称 / 混合输入解析
- 多选门店增值模块
- 多选总部模块
- 基于 `chat_id + user_id` 的临时会话状态保存
- 最终调用现有报价脚本并回传摘要与文件

### 本次不包含

- 飞书卡片按钮回调
- 飞书原生下拉、多选控件回调
- 人工审批流
- 非标准商务场景自动化
- 把利润写入对客报价单

## 用户体验

### 交互原则

- 每一步只问一个问题
- 卡片里给出明确候选项和回复示例
- 用户既可以回复数字，也可以回复中文名称
- 已经识别出的信息尽量自动吸收，不重复追问
- 用户随时可回复 `重来`、`取消`

### 典型流程

1. 用户说“开始报价”或直接说自然语言需求
2. 机器人发送起始卡片，展示当前流程说明
3. 机器人逐步询问：
   - 品牌名称
   - 餐饮类型
   - 门店数量
   - 套餐
   - 门店增值模块
   - 总部模块
   - 配送中心数量（如需）
   - 生产加工中心数量（如需）
4. 机器人发送确认卡片，展示配置摘要
5. 用户回复 `生成报价` 或 `确认`
6. 机器人生成报价，并发送：
   - 摘要文本
   - PDF 文件
   - Excel 文件
   - JSON 文件

### 卡片交互格式

每一步卡片采用统一结构：

- 标题：当前步骤名称
- 当前问题：本轮只收集一个字段
- 选项列表：编号 + 中文名称
- 已选摘要：避免用户忘记上下文
- 回复示例：数字、中文、混合示例

示例：

- 标题：`第 4 步：选择门店增值模块`
- 已选摘要：`正餐 / 10 店 / 旗舰版`
- 回复示例：
  - `1,3`
  - `厨房KDS, 成本管理`
  - `1, 成本管理`
  - `无`

## 方案对比

### 方案 A：纯文本引导

- 优点：实现最简单
- 缺点：体验弱，难体现“卡片化引导”

### 方案 B：静态卡片 + 文本回复

- 优点：不依赖回调，兼顾体验与实现复杂度
- 缺点：不是真正点击提交，仍需用户发送一条文本

### 方案 C：单张大卡片一次性填报

- 优点：轮次少
- 缺点：首屏信息过重，解析复杂，出错后难纠正

### 结论

采用方案 B。它最符合当前飞书接入限制，也最接近“少打字、以选择为主”的目标。

## 系统设计

### 新增能力分层

1. 会话状态层
   - 保存当前会话处于哪个步骤
   - 保存已收集的表单数据
   - 保存最近一次卡片上下文

2. 输入解析层
   - 将用户文本解析为当前步骤对应的结构化值
   - 支持数字、中文、混合输入
   - 支持多选分隔符解析

3. 卡片渲染层
   - 根据当前步骤生成飞书静态卡片
   - 同时输出已选摘要和回复示例

4. 流程编排层
   - 决定下一步问什么
   - 根据已选模块决定是否追问数量
   - 满足条件后进入生成报价

5. 报价执行层
   - 复用现有配置生成与文件生成逻辑

### 推荐新增模块

- `scripts/feishu_quote_session.py`
  - 会话状态读写
  - 过期清理

- `scripts/feishu_quote_parser.py`
  - 当前步骤输入解析
  - 多选解析
  - 同义词和容错匹配

- `scripts/feishu_quote_cards.py`
  - 飞书卡片 JSON 组装
  - 各步骤卡片模板

- `scripts/feishu_quote_flow.py`
  - 多轮流程推进
  - 状态迁移
  - 与报价脚本衔接

现有 `scripts/feishu_file_delivery.py` 保持负责消息与文件发送，不承担业务编排。

## 数据模型

### 会话状态

建议字段：

- `session_id`
- `chat_id`
- `user_id`
- `current_step`
- `form_data`
- `last_card_type`
- `updated_at`
- `expires_at`

### form_data

建议字段：

- `客户品牌名称`
- `餐饮类型`
- `门店数量`
- `门店套餐`
- `门店增值模块`
- `总部模块`
- `配送中心数量`
- `生产加工中心数量`

### 存储方式

MVP 使用本地文件存储。

- 路径建议：`data/feishu_quote_sessions/`
- 每个会话一个 JSON 文件
- 文件名建议：`{chat_id}__{user_id}.json`
- 默认 TTL：24 小时

原因：

- 当前需求量级较小
- 无需引入数据库
- 易于排查
- 与现有脚本式项目结构匹配

## 状态流转

### 步骤定义

1. `await_brand_name`
2. `await_meal_type`
3. `await_store_count`
4. `await_package`
5. `await_store_modules`
6. `await_hq_modules`
7. `await_delivery_center_count`
8. `await_production_center_count`
9. `await_confirm_generate`
10. `completed`

### 跳步规则

- 若自然语言中已识别出品牌名、餐饮类型、门店数、套餐，则直接跳过已明确步骤
- 若总部模块中未选择 `配送中心`，跳过 `await_delivery_center_count`
- 若总部模块中未选择 `生产加工`，跳过 `await_production_center_count`
- 若套餐未指定，默认填入 `旗舰版`

## 输入解析设计

### 支持输入格式

- 单选数字：`1`
- 多选数字：`1,3,5`
- 中文名称：`正餐`
- 多选中文：`厨房KDS, 成本管理`
- 混合输入：`1, 成本管理`
- 跳过输入：`无`、`不选`
- 控制命令：`重来`、`取消`

### 解析规则

- 多选统一支持逗号、中文逗号、顿号、空格拆分
- 数字按当前卡片选项序号映射
- 中文名称按候选项精确匹配优先，必要时做小范围别名匹配
- 不允许跨餐饮类型选错模块
- 输入解析失败时，返回纠错卡片，不丢失已有状态

### 容错策略

- `kds` 可匹配 `厨房KDS`
- `scrm` 可匹配 `企业微信SCRM`
- `配送` 可提示候选 `配送中心`
- 模糊度过高时不给猜测结果，直接要求用户重选

## 卡片设计

### 卡片类型

- 欢迎卡
- 单选卡
- 多选卡
- 数量输入提示卡
- 确认生成卡
- 纠错卡
- 成功卡
- 转人工卡

### 卡片内容要求

- 纯展示，不依赖卡片按钮回调
- 显示当前步骤与总流程位置
- 显示已选摘要
- 给出回复示例
- 控制文案长度，适合飞书聊天阅读

## 业务规则

### 适用范围

- `1~30 店` 进入标准自动报价
- `31 店及以上` 直接转人工

### 默认规则

- 未指定套餐时，默认 `旗舰版`
- 未选择门店增值模块时，按空列表处理
- 未选择总部模块时，按空列表处理

### 模块规则

- 门店增值模块支持多选
- 总部模块支持多选
- 只有选中 `配送中心` 才追问配送中心数量
- 只有选中 `生产加工` 才追问生产加工中心数量

## 错误处理

### 用户输入错误

- 不返回技术堆栈
- 返回纠错卡片
- 明确指出当前未识别的输入
- 重列合法候选项

### 业务边界错误

- 超过 30 店：返回转人工卡片
- 非标准场景：返回转人工提示
- 缺少关键字段：继续收集，不提前生成

### 系统错误

- 发送一条简洁错误消息
- 保留会话状态，便于用户重试
- 记录本地日志，便于排查

## 风险与缓解

### 风险 1：文本解析歧义

缓解：

- 限制候选项数量
- 当前步骤只解析当前字段
- 模糊输入不强猜

### 风险 2：会话状态串话

缓解：

- 使用 `chat_id + user_id` 做唯一键
- 每次更新都回写 `updated_at`

### 风险 3：卡片过长影响阅读

缓解：

- 长模块列表按餐饮类型拆分
- 每步只展示当前必要选项

### 风险 4：MVP 与 OpenClaw 交互风格不一致

缓解：

- 保持问题顺序和字段口径一致
- 共用同一套表单字段名与业务规则

## 实施边界

本次实现只解决：

- 飞书场景中如何少打字完成报价
- 不依赖回调的多轮引导
- 与现有报价引擎的可靠对接

本次不解决：

- 真正按钮点击即提交
- 飞书审批或卡片回写
- 跨平台统一 UI 引擎

## 成功标准

- 用户在飞书里无需长句输入即可完成标准报价
- 用户可用数字、中文或混合方式完成选择
- 多选模块能稳定识别
- 机器人能在流程结束后成功输出报价文件
- 超范围场景能清晰转人工，而不是卡死或误报价
